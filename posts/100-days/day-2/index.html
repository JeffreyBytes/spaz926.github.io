Day 2 of 100: Started beefing up my knowledge on closures. Have always found this subject to be fairly bloated and nonintuitive, but I’m sure with enough practice it’ll become second nature.

Closures as Parameters

Can’t say I’m too fond of the idea of using () -> Void in my code. It looks very… nonSwifty. You’d think there would be a better syntax when your closure has no parameters or return values.

When the closure accepts a parameter and returns a value the syntax makes much more sense to me, but stil feels less Swifty than the rest of the language.

Trailing Closure Syntax

Now this is starting to click, and I can finally read some more advanced Swift code and understand what is going on.

Shorthand Parameter Names

The usage of shorthand parameter names in closures ($0, $1, $2) reminds me a lot of writing shell scripts for bash, and I think I’ll be using them often in my adventures with Swift.

Capturing Values

I hate to admit it, but this took me for a bit of a loop. I was having trouble wrapping my mind around the idea of capturing. Once the lightbulb turned on in my head, I felt stupid that something so simple took so long to understand.

Basically if a closure uses a value defined outside of its scope it captures that value so it can use it. Below is the example from Hacking with Swift.

func travel() -> (String) -> Void {
	var counter = 1
	
	return {
		print("\(counter). I'm going to \($0).")
		counter += 1
	}
}

In the above example since counter is defined outside of the scope of the closure, the closure captures the value so it can use it.

This means that we can set a constant, result, equal to travel() to get back the closure. Then call the closure freely and it will still have access to the counter variable.

let result = travel()
result("London") // Prints "1. I'm going to London."

And if we call result("London") multiple times the counter will increment.

result("London") // Prints "2. I'm going to London."
result("London") // Prints "3. I'm going to London."
result("London") // Prints "4. I'm going to London."

Final Thoughts For the Day

I’m definitely going to have to continue studying closures before they make complete sense, but thanks to the Hacking with Swift simple approach to the subject I’m feeling more confident in my knowledge of the subject.

If you’re also taking part in Hacking with Swift’s 100 Days of Swift, I highly suggest checking out the official Swift Language Guide and follow along. It is a great suppliment to the lessons on the website. It really helped me solidify some concepts today.

You’ll probably notice I’m not following the exact path of the 100 Days of Swift challenge. That’s because I feel comfortable in my abilities to not need to study up on simple data types, loops, etc. I have been working with Swift on-and-off since it’s inception, and have been coding in some fasion since I was in middle school.
